# Playwright-Python OrangeHRM Framework Blueprint & Style Guide

This document serves as the absolute, comprehensive architectural blueprint and coding style guide for the Playwright-Python automation framework. It outlines the core directory structure, naming standards, and advanced engineering patterns developed to guarantee high reliability, zero-flakiness, and seamless hybrid UI-API verification.

---

## 1. Repository Architecture

The project implements a strict Page Object Model (POM) with a clean separation of concerns:
* **`pages/`**: Page Object classes containing component interactions and assertions. Every page class inherits from `BasePage`.
* **`locators/`**: Static locator selectors (stored as standard string CSS or XPath selectors) partitioned by page/component.
* **`constants/`**: Centralized test data, enums, endpoints, and default/test configurations.
* **`tests/`**: Pytest test cases organized by module (e.g., `tests/orangehrm/components/`).
* **`api/`**: API client utilities (inheriting from `BaseAPI`) for hybrid UI-API end-to-end integration.

---

## 2. Locator Conventions & Style Rules

OrangeHRM uses consistent design tokens across all components. Brittle selectors must be avoided. Always adhere to these DOM class structures:

### A. Button Design Tokens
* **Primary Branding (Orange Buttons)**: Used for active actions like **Search**, **Save**, and **Add**. These use the `.oxd-button--secondary` class, NOT `--primary`.
  * *Search Button*: `"button.oxd-button--secondary:has-text('Search')"`
  * *Add Button*: `"button.oxd-button--secondary:has-text('Add')"`
* **Secondary Branding (Grey/White Buttons)**: Used for outline/neutral actions like **Cancel** and **Reset**. These use the `.oxd-button--ghost` class.
  * *Reset Button*: `"button.oxd-button--ghost:has-text('Reset')"`

### B. Grid & List Layouts
* **Table Grid Container**: OrangeHRM lists are composed entirely of CSS-styled `div`s; there are no native `<table>` elements. Do not combine tags with classes like `table.oxd-table-card` (which will fail). The standard locator for results grids is:
  * *Results Grid*: `".oxd-table"`
* **Empty-State Text**: To verify "No Records Found" reliably, avoid brittle layout class names (e.g., `.orangehrm-empty-message-container` which can be absent or delayed). Use a bulletproof text-based XPath:
  * *No Records Text*: `"//span[text()='No Records Found']"`

### C. Form Inputs & Select Dropdowns
* **Dropdown Box Container**: Dropdown boxes use the CSS container class `.oxd-select-text`, NOT `.oxd-select-text-container`.
  * *Sub Unit Dropdown*: `"//label[contains(text(), 'Sub Unit')]/following::div[contains(@class, 'oxd-select-text')]"`
* **Dropdown Selection Options**: Option elements inside an active overlay use the class `.oxd-select-option` (which changes dynamically to `.oxd-select-option --selected` when highlighted). Target these robustly using:
  * *Dropdown Option Option*: `lambda option: f"//div[contains(@class, 'oxd-select-option') and contains(., '{option}')]"`

## 3. High-Reliability Engineering Patterns

To ensure test suites achieve complete sequential stability with zero retries, the following design patterns are strictly enforced:

### A. The "Import-Time Constants Evaluation" Gotcha
**The Concern**: Defining dynamic variables (like time epochs or random names) as static class variables in a module leads to single-evaluation at import time. Sequential parameterized runs or test retries within the same process will reuse the exact same stale value.
* **The Solution**: Centralize configurations, but turn dynamic variables into dynamic properties or class methods:
  ```python
  class EmergencyContactsTestData:
      RELATIONSHIP = "Brother"
      
      @classmethod
      def get_dynamic_contact_name(cls) -> str:
          return f"Contact_{int(time.time())}"
  ```

### B. Preventing Cancelled Network Requests (`Protocol error`)
**The Concern**: Clicking sidebar options (like "Recruitment" or "Leave") already defaults to loading the default landing sub-pages (Candidates list and Leave List). Clicking a secondary top-navigation accordion link immediately afterward triggers a second duplicate navigation that aborts the first request, leading to Playwright throwing `Protocol error (Network.getResponseBody): No resource with given identifier found`.
* **The Solution**: Keep `navigate_via_menu()` highly streamlined. Only perform the top-navigation click if it is functionally different from the default landing page. Remove redundant navigation accordion clicks.

### C. Autocomplete "No-Op" & Overlay Blur
**The Concern**: OrangeHRM's autocomplete input fields (such as `Candidate Name` or `Employee Name`) ignore searches that do not match a valid suggestion inside the overlay. Clicking "Search" on an invalid autocomplete input acts as a no-op (the browser blocks the submit, meaning no network call is sent), causing `expect_response` timeouts.
* **The Solution**:
  1. For non-existent filtering tests, **avoid autocomplete fields**. Filter by standard text inputs instead (e.g., filter by `keywords` on Candidates, or by an ancient date range on Leave List). These are guaranteed to trigger the search API and return empty results.
  2. Implement robust overlay blurring in `autocomplete_dropdown()`. If the suggestion is not found, catch the timeout and trigger an `Escape` keypress to close the active dropdown overlay and return focus:
     ```python
     try:
         expect(target_option).to_be_visible(timeout=3000)
         target_option.click()
     except AssertionError:
         print(f"Option '{text}' not found, proceeding without selecting.")
         autocomplete_input.press("Escape")
     ```

### D. Dynamic API Response Interception & Mapping
**The Pattern**: Page Object action methods (like `filter_leaves()` or `filter_candidates()`) must intercept and return the active filtered API response payload. This allows test cases to validate UI elements against the exact data returned by that specific action:
```python
def filter_leaves(self, **kwargs):
    with pulse_step("Filtering leaves"):
        # Fill fields...
        response = self.base_page.wait_for_api_call(
            lambda: self.click(LeaveListLocators.SEARCH_BUTTON),
            LeaveListConstants.LEAVE_REQUESTS_URL
        )
        self.base_page.wait_for_fully_page_loaded()
        return response
```

### E. Adaptive UI Assertions
**The Pattern**: Never hardcode positive assertions (like expecting a grid table to be visible) if the database state is dynamic. Assertions must evaluate the returned API response count dynamically to determine the expected UI state:
```python
def verify_records_exist(self, response):
    with pulse_step("Verifying records"):
        if response["meta"]["total"] > 0:
            self.verify_element_is_visible(LeaveListLocators.RESULTS_TABLE)
            self.verify_element_is_not_visible(LeaveListLocators.NO_RECORDS_TEXT)
        else:
            self.verify_element_is_visible(LeaveListLocators.NO_RECORDS_TEXT)
            self.verify_element_is_not_visible(LeaveListLocators.RESULTS_TABLE)
```

### F. Capturing Baseline State Before Actions
**The Pattern**: To verify form filters reset correctly, do not hardcode comparisons against arbitrary expected values (which can change based on the system's dynamic date or region). Instead, capture the baseline input value *before* applying the filter, execute the filter and reset actions, and verify that the field returns to the captured baseline:
```python
# Capture baseline
baseline_date = page.locator(LeaveListLocators.FROM_DATE).get_attribute("value") or ""

# Filter and Reset
leave_list_page.filter_leaves(from_date=LeaveListConstants.TEST_DATE)
leave_list_page.reset_filters()

# Assert against captured baseline
leave_list_page.verify_date_reset(LeaveListLocators.FROM_DATE, baseline_date)
```

## 4. Test Framework Integration (Pulse & Pytest)

* **Severity & Tags**: Annotate test methods with `@pytest.mark.pulse_severity()` and `@pytest.mark.pulse_tag()`.
* **Pulse Steps**: High-level test actions inside test methods and Page Object methods must be wrapped in `with pulse_step("Description"):` context managers or decorated with `@step("Description")` to ensure accurate reporting.
* **No `networkidle` state**: Rely strictly on `self.page.wait_for_load_state("domcontentloaded")` and `self.page.wait_for_load_state("load")` to prevent flaky timeouts on public assets.
